# 신뢰성 감사 보고서

작성일: 2026-07-15  
대상: Vite React SPA 프런트엔드  
목적: FastAPI 응답과 Gemini 생성 주식 콘텐츠를 소비하는 화면의 신뢰성 위험을 확정 증거 기준으로 정리한다.

## 1. 범위와 방법

### 검토 범위

* 프런트엔드의 라우팅, 필터 정규화, API 오류 표시, 상세 이동, 배치 화면 상태 관리, 외부 링크 렌더링, Gemini/API DTO 표시 안정성
* 읽기 전용 API 계약: `docs/api-spec.json`
* 사용자 제공 기준 검증 결과: `pnpm test`, `pnpm lint`, `pnpm build`, `pnpm audit --prod`

### 제외 범위

* 백엔드 구현 변경
* Gemini 프롬프트, 검증, 그라운딩, 재시도 정책의 실제 수정
* 새 테스트 실행 또는 새 테스트 결과 주장
* `docs/api-spec.json`, `src` 파일, 패키지 파일, git 상태 변경

### 방법

1. 사용자 제공 기준 검증 결과를 기준 상태로 기록했다.
2. `docs/api-spec.json`을 읽기 전용 API 계약으로 확인했다.
3. 이미 확정된 프런트엔드 신뢰성 결함만 발견 항목에 포함했다.
4. 아직 적용되지 않은 수정은 완료로 표시하지 않았다.

## 2. 기준 검증 상태

다음 결과는 사용자 제공 컨텍스트에 따른 기준 상태다. 이 보고서 작성 과정에서 새로 실행한 검증 결과가 아니다.

* `pnpm test`: 39개 중 39개 통과
* `pnpm lint`: 통과
* `pnpm build`: 통과
* `pnpm audit --prod`: 통과

해석: 현재 기준 검증은 통과했지만, 아래 결함은 정적 계약, 화면 상태, 입력 정규화, 오류 처리, 링크 안전성, DTO 렌더링의 의미적 신뢰성 문제다. 테스트 통과만으로 사용자 영향이 사라졌다고 볼 수 없다.

## 3. 읽기 전용 API 계약 출처

`docs/api-spec.json`은 감사 근거로만 사용했다. 이 파일은 사용자 제공 컨텍스트상 추적되지 않은 파일이며, 커밋 대상에 포함하면 안 된다.

계약상 확인된 주요 근거:

* 배치 목록 API는 `fromDate`, `toDate`, `status`, `page`, `size`를 쿼리로 받으며 `page`는 최소 1이다.
* 아카이브 API의 `status`는 `READY`, `PARTIAL`, `FAILED` 중 하나이거나 null이다.
* 일자 기반 페이지 조회는 `businessDate`와 선택적 `versionNo`를 사용한다.
* ID 기반 페이지 조회는 `pageId`를 사용한다.
* FastAPI 검증 실패는 422 응답이며 `HTTPValidationError.detail` 배열을 반환한다.
* 뉴스 기사 DTO에는 `originLink`, `naverLink`, `sourceSummary`가 포함될 수 있다.
* Gemini/API 생성 콘텐츠는 `summary.short`, `summary.long`, `summary.analysis`, `globalHeadline`, `markets`, `metadata` 등 여러 중첩 필드로 렌더링된다.

## 4. 확정 발견 사항

### 4.1 기본 날짜 범위 역전

심각도: 높음

증거:

* 배치와 아카이브 계열 API는 `fromDate`와 `toDate`를 별도 쿼리로 받는다.
* 확정 결함으로 기본 날짜 범위가 시작일과 종료일이 뒤집힌 상태로 전달될 수 있음이 확인됐다.

영향:

* 기본 필터만 사용해도 빈 목록, 잘못된 기간 조회, 백엔드 422 또는 의미 없는 결과가 발생할 수 있다.
* 운영자가 배치 실패나 누락을 실제보다 없거나 다르게 판단할 위험이 있다.

수정 방향:

* 기본값 생성 시 항상 `fromDate <= toDate`를 보장한다.
* URL 파싱, 폼 초기값, API 파라미터 변환 지점에 동일한 날짜 정규화 함수를 적용한다.
* 역전 입력을 받으면 자동 교정하거나 사용자에게 명확히 표시한다.

### 4.2 잘못된 페이지와 상태 정규화

심각도: 높음

증거:

* API 계약에서 `page`는 최소 1이다.
* 아카이브 `status`는 `READY`, `PARTIAL`, `FAILED` 또는 null만 허용한다.
* 확정 결함으로 잘못된 페이지 값과 상태 값이 정상 값처럼 남아 요청 또는 화면 상태에 반영될 수 있음이 확인됐다.

영향:

* `page=0`, 음수, 숫자가 아닌 값이 목록 상태를 깨뜨릴 수 있다.
* 계약에 없는 상태 문자열이 필터에 남으면 422, 빈 결과, 잘못된 배지 표시가 발생할 수 있다.
* URL 공유 시 같은 링크를 열어도 사용자마다 다른 화면으로 보일 수 있다.

수정 방향:

* URL에서 들어온 `page`는 정수화 후 최소 1로 보정한다.
* `status`는 허용 목록 기반으로만 유지하고, 그 외 값은 제거한다.
* 정규화된 값으로 URL, 폼 상태, API 요청을 같은 방향으로 맞춘다.

### 4.3 422 오류 상세 처리 부족

심각도: 중간

증거:

* API 계약은 422 응답에서 `HTTPValidationError.detail` 배열을 반환한다.
* 확정 결함으로 이 상세 배열이 사용자 또는 개발자에게 충분히 보존되어 표시되지 않을 수 있음이 확인됐다.

영향:

* 잘못된 필터나 요청 본문이 있을 때 원인을 알 수 없는 일반 오류로 보일 수 있다.
* 운영자는 어떤 필드가 잘못됐는지 파악하기 어렵고, 재시도나 수정 행동이 늦어진다.

수정 방향:

* 422 응답의 `detail` 배열을 안전하게 파싱한다.
* 필드 경로, 메시지, 오류 타입을 사람이 읽을 수 있는 문장으로 표시한다.
* 알 수 없는 오류와 검증 오류를 구분해 로그와 UI 메시지를 분리한다.

### 4.4 아카이브 상세 이동의 식별자와 버전 혼선

심각도: 높음

증거:

* API 계약은 일자 기반 조회에 `businessDate`와 선택적 `versionNo`를 사용한다.
* ID 기반 조회는 `pageId`를 사용한다.
* 확정 결함으로 아카이브 화면에서 상세 이동 시 식별자와 버전 기준이 일관되지 않을 수 있음이 확인됐다.

영향:

* 사용자가 선택한 아카이브 항목이 아닌 최신판 또는 다른 버전의 페이지로 이동할 수 있다.
* 감사, 재현, 운영 확인이 필요한 과거 일자 페이지에서 신뢰성이 떨어진다.

수정 방향:

* 아카이브 목록 항목의 고유 식별 기준을 하나로 정한다.
* `pageId`가 있으면 ID 기반 이동을 우선하고, 일자 기반 이동은 `businessDate`와 `versionNo`를 함께 전달한다.
* 링크 생성과 상세 조회 훅에서 같은 식별자 정책을 사용한다.

### 4.5 소스 보기 상태가 `READY`로 하드코딩됨

심각도: 중간

증거:

* API 계약상 페이지와 아카이브 상태는 `READY`, `PARTIAL`, `FAILED` 등 실제 상태를 가질 수 있다.
* 확정 결함으로 소스 보기 계열 UI에서 상태가 실제 DTO 대신 `READY`로 고정 표시될 수 있음이 확인됐다.

영향:

* 부분 생성 또는 실패 상태의 콘텐츠가 정상 완료처럼 보인다.
* Gemini 생성 콘텐츠의 품질 확인, 운영 판단, 사용자 신뢰에 영향을 준다.

수정 방향:

* 표시 상태는 반드시 API DTO의 실제 `status` 또는 파생된 안전 상태에서 가져온다.
* 상태가 없거나 알 수 없으면 `READY`로 대체하지 말고 별도 미확인 상태를 표시한다.

### 4.6 배치 선택 상태가 오래 남고 상세 숨김 상태가 꼬임

심각도: 중간

증거:

* 배치 목록과 상세 API는 목록 항목, 페이지네이션, 요약, 상세를 별도 데이터로 제공한다.
* 확정 결함으로 선택한 배치가 목록 변경 뒤에도 오래 남거나, 상세 패널 숨김 상태와 선택 상태가 불일치할 수 있음이 확인됐다.

영향:

* 사용자가 현재 목록에 없는 작업의 상세를 보고 있다고 착각할 수 있다.
* 필터 변경, 페이지 이동, 재조회 뒤 운영 판단이 잘못될 수 있다.

수정 방향:

* 목록 쿼리 키가 바뀌면 선택된 작업과 상세 표시 상태를 재검증한다.
* 선택된 작업이 현재 목록에 없으면 선택을 해제하거나 명시적으로 과거 선택임을 표시한다.
* 상세 패널 열림 상태와 선택 ID를 단일 상태 전이로 관리한다.

### 4.7 외부 URL 렌더링이 안전하지 않음

심각도: 높음

증거:

* 뉴스 기사 DTO는 `originLink`와 `naverLink`를 문자열로 제공한다.
* 확정 결함으로 외부 URL이 안전한 프로토콜과 새 창 보안 속성 검증 없이 렌더링될 수 있음이 확인됐다.

영향:

* `javascript:` 같은 위험한 URL 또는 변형된 외부 링크가 클릭 가능해질 수 있다.
* 새 창 링크에서 `window.opener` 기반 탭 탈취 위험이 생길 수 있다.
* Gemini 또는 외부 수집 데이터가 그대로 링크가 되는 구조에서는 방어가 필수다.

수정 방향:

* `http:`와 `https:`만 허용하는 URL 정규화 함수를 둔다.
* 외부 링크에는 `target="_blank"` 사용 시 `rel="noopener noreferrer"`를 함께 적용한다.
* 검증 실패 URL은 링크가 아닌 텍스트 또는 비활성 상태로 렌더링한다.

### 4.8 malformed Gemini/API DTO 렌더링 취약성

심각도: 높음

증거:

* API 계약은 Gemini/API 생성 콘텐츠를 여러 중첩 DTO로 반환한다.
* `summary.short`, `summary.long`, `summary.analysis`, `globalHeadline`, `markets`, `metadata`, 기사 링크와 요약 등 null 가능 필드와 배열 필드가 섞여 있다.
* 확정 결함으로 형태가 깨진 DTO가 들어올 때 렌더링이 깨지거나 잘못된 내용이 표시될 수 있음이 확인됐다.

영향:

* Gemini 응답 일부가 누락되거나 형식이 어긋난 경우 전체 화면이 비거나 런타임 오류가 날 수 있다.
* 잘못된 숫자, 날짜, 요약, 기사 목록이 정상 정보처럼 보일 수 있다.
* 주식 콘텐츠 특성상 잘못된 렌더링은 투자 판단 보조 정보의 신뢰를 떨어뜨린다.

수정 방향:

* DTO 경계에서 런타임 파싱과 기본값 정책을 둔다.
* 배열, null 가능 문자열, 날짜 문자열, 숫자 문자열을 표시 전 안전하게 정규화한다.
* 필수 섹션이 깨졌을 때 화면 전체 실패 대신 섹션 단위 오류 상태를 표시한다.

## 5. 완료된 수정 기록

이 섹션은 최초 감사 이후 프런트엔드에서 실제 적용되고 검증된 수정만 기록한다. 백엔드 전용 Gemini 그라운딩, 검증, 재시도 정책과 배치 권한 검증은 아직 해결된 것으로 표시하지 않는다.

### 5.1 입력 정규화 수정 완료

상태: fixed

적용 내용:

* 날짜 범위 입력은 요청 전 `fromDate <= toDate`가 되도록 정규화했다.
* URL에서 들어온 `page`는 정수화하고 최소 1로 보정했다.
* 아카이브 `status`는 계약상 허용 값인 `READY`, `PARTIAL`, `FAILED`만 유지하고, 그 외 값은 제거하도록 정리했다.
* 정규화된 값이 URL 상태, 화면 상태, API 요청 파라미터에 같은 방식으로 반영되도록 맞췄다.

관련 검증:

* 프런트엔드 테스트 스위트에 입력 정규화 회귀 테스트가 포함됐다.
* 최신 확인된 전체 테스트 스위트 결과는 58개 테스트 통과다.
* `pnpm lint`와 `pnpm build`는 수정 흐름에서 통과가 확인됐다.

### 5.2 422 검증 오류 표시 수정 완료

상태: fixed

적용 내용:

* FastAPI 422 응답의 `HTTPValidationError.detail` 배열을 보존해 파싱하도록 수정했다.
* 필드 경로, 메시지, 오류 타입을 사람이 읽을 수 있는 오류 문장으로 표시하도록 했다.
* 검증 오류와 일반 API 오류를 구분해 사용자가 잘못된 입력 원인을 확인할 수 있게 했다.

관련 검증:

* 422 상세 배열 표시와 일반 오류 fallback에 대한 회귀 테스트가 추가 또는 갱신됐다.
* 최신 확인된 전체 테스트 스위트 결과는 58개 테스트 통과다.
* `pnpm lint`와 `pnpm build`는 수정 흐름에서 통과가 확인됐다.

### 5.3 아카이브 상세 이동 수정 완료

상태: fixed

적용 내용:

* 아카이브 목록 항목에 `pageId`가 있으면 ID 기반 상세 이동을 우선하도록 했다.
* `pageId`가 없는 기존 링크는 기존의 `businessDate` 기반 조회 동작을 유지하도록 했다.
* 잘못된 `pageId` 또는 잘못된 상세 라우트 입력은 유효한 상세 요청으로 진행하지 않도록 방어했다.

관련 검증:

* `pageId` 우선 이동, 기존 날짜 기반 fallback, 잘못된 상세 라우트 방어에 대한 회귀 테스트가 포함됐다.
* 최신 확인된 전체 테스트 스위트 결과는 58개 테스트 통과다.
* `pnpm lint`와 `pnpm build`는 수정 흐름에서 통과가 확인됐다.

### 5.4 소스 상태 전파 수정 완료

상태: fixed

적용 내용:

* 소스 보기 계열 UI가 상태를 `READY`로 고정하지 않고 API DTO의 실제 상태를 사용하도록 수정했다.
* `PARTIAL`, `FAILED`, 미확인 상태가 정상 완료처럼 표시되지 않도록 상태 표시 경로를 정리했다.

관련 검증:

* 실제 소스 상태가 화면에 전파되는지 확인하는 회귀 테스트가 포함됐다.
* 최신 확인된 전체 테스트 스위트 결과는 58개 테스트 통과다.
* `pnpm lint`와 `pnpm build`는 수정 흐름에서 통과가 확인됐다.

### 5.5 배치 상세 상태 관리 수정 완료

상태: fixed

적용 내용:

* 필터, 페이지, 목록 결과가 바뀔 때 현재 선택된 배치가 새 목록에 남아 있는지 재검증하도록 했다.
* 현재 목록에 없는 선택 항목은 상세 패널과 함께 정리해 오래된 상세가 남지 않도록 했다.

관련 검증:

* 오래된 선택 해제, 상세 패널 숨김, 상세 로딩과 오류 초기화에 대한 회귀 테스트가 포함됐다.

### 5.9 명시적 잔여 항목

상태: residual

남은 위험:

* Gemini 생성 결과의 백엔드 스키마 검증, 그라운딩, 재시도와 부분 실패 처리 정책은 이 프런트엔드 수정으로 해결되지 않았다.
* 배치 실행과 조회 계열 API의 실제 권한 검증, 역할 제한, 감사 로그는 백엔드에서 해결해야 한다.
* 프런트엔드는 방어적 표시와 입력 정규화를 적용했지만, 서버 측 데이터 무결성과 권한 보장은 여전히 백엔드 책임이다.

## 6. 백엔드 전용 잔여 위험

다음 항목은 프런트엔드만으로 완전히 해소할 수 없는 잔여 위험이다. 프런트엔드는 방어적 표시를 해야 하지만, 근본 보장은 백엔드에서 필요하다.

### 6.1 Gemini 검증 위험

* Gemini 생성 결과가 계약에 맞는지 백엔드에서 스키마 검증해야 한다.
* 누락 필드, 빈 배열, 잘못된 날짜, 잘못된 링크, 의미 없는 요약을 저장 전 거부하거나 보정해야 한다.

### 6.2 Gemini 그라운딩 위험

* 생성된 요약과 분석이 실제 기사 또는 원천 데이터에 근거하는지 백엔드에서 추적 가능해야 한다.
* 프런트엔드는 출처를 보여줄 수 있지만, 생성 내용과 출처의 대응 관계 검증은 백엔드 책임이다.

### 6.3 Gemini 재시도와 부분 실패 위험

* 모델 호출 실패, 부분 응답, 시간 초과, 재시도 뒤 중복 생성이 발생할 수 있다.
* 백엔드는 멱등성, 재시도 한도, 부분 성공 상태, 실패 사유를 명확히 기록해야 한다.

### 6.4 배치 권한 위험

* API 계약상 배치 실행과 조회 계열 엔드포인트는 `authorization` 헤더를 받는다.
* 실제 권한 검증, 역할 제한, 강제 재실행 권한, 감사 로그는 백엔드에서 보장해야 한다.
* 프런트엔드 버튼 비활성화만으로는 배치 실행 권한을 보호할 수 없다.

## 7. 다음 수정 우선순위

1. URL과 필터 정규화: 날짜 범위, 페이지, 상태를 먼저 고친다.
2. API 오류 처리: 422 상세 배열을 보존하고 표시한다.
3. 상세 이동 정책: 아카이브 식별자와 버전 이동을 통일한다.
4. 외부 링크 방어: URL 프로토콜과 새 창 보안 속성을 강제한다.
5. DTO 방어 렌더링: Gemini/API 응답을 섹션 단위로 안전하게 표시한다.
6. 배치 상태 관리: 선택 ID와 상세 패널 상태를 목록 변화에 맞춰 재검증한다.
