# PRD 초안

**프로젝트명:** 미국/한국 주식 시장 일간 뉴스 요약 페이지
**문서 버전:** v0.1 Draft
**문서 목적:** 개발자, 디자이너, 기획자가 동일한 기준으로 구현 범위와 산출물을 이해할 수 있도록 기능/화면/제약 사항을 명확히 정의한다.

---

# 1. Project Overview

## 1-1. 프로젝트 배경

개인 투자 또는 시장 모니터링을 목적으로 미국 및 한국 주식 시장의 주요 흐름을 매일 빠르게 파악하고자 할 때, 사용자는 여러 뉴스 기사와 지수 데이터를 개별적으로 확인해야 한다. 이 과정은 시간이 많이 소요되며, 동일 이슈의 중복 기사와 과도한 정보량 때문에 핵심 내용을 빠르게 파악하기 어렵다.

본 프로젝트는 **미국/한국 주식 시장의 대표 지수 정보와 주요 뉴스들을 일별로 수집·저장·정리하여, 하나의 일간 페이지로 제공하는 개인용 웹 서비스**를 구축하는 것을 목표로 한다.

## 1-2. 해결하려는 문제

현재 사용자가 겪는 핵심 문제는 다음과 같다.

- 미국/한국 증시 관련 뉴스를 여러 사이트에서 직접 찾아야 한다.
- 동일 이슈가 여러 언론사에서 반복 보도되어 정보 중복이 심하다.
- 전일 시장 흐름을 한 번에 정리한 페이지가 없어 빠른 복기가 어렵다.
- 시장 지수, 뉴스, 요약, 원문 링크가 분리되어 있어 맥락 파악이 불편하다.

## 1-3. 프로젝트 범위

본 프로젝트의 1차 범위는 아래와 같다.

| 구분        | 범위 포함                           | 범위 제외                         |
| ----------- | ----------------------------------- | --------------------------------- |
| 시장 범위   | 미국, 한국                          | 개별 국가 추가 확장               |
| 뉴스 수집   | Naver 뉴스 검색 API 기반 수집       | 직접 HTML 크롤링 기반 대규모 수집 |
| 지수 수집   | Yahoo Finance 기반 대표 지수 수집   | 개별 종목 시세/차트 고도화        |
| 저장소      | MariaDB                             | 별도 검색엔진(OpenSearch 등)      |
| 배치 실행   | API 엔드포인트 + cronjob(curl 호출) | 복잡한 워크플로우 오케스트레이터  |
| 페이지 유형 | 일자별 시장 요약 페이지             | 실시간 스트리밍 뉴스 대시보드     |
| 사용자 범위 | 개인 사용                           | 다중 사용자 권한/계정 시스템      |

## 1-4. 핵심 가치 제안

- 전일 시장 흐름을 **하나의 페이지**에서 빠르게 복기 가능
- 대표 지수와 뉴스가 **같은 맥락**으로 묶여 제공됨
- 중복 기사 제거 및 요약을 통해 **핵심 정보만 소비**
- 과거 일자 데이터 보관으로 **일별 비교 및 회고 가능**

# 2. Goals & Success Metrics

## 2-1. 핵심 목표

1. 미국/한국 시장별로 **일자 단위 요약 페이지**를 자동 생성한다.
2. 대표 지수 정보와 주요 뉴스 요약, 원문 링크를 함께 제공한다.
3. 중복 뉴스 제거 및 AI 요약을 통해 사용자 정보 소비 시간을 줄인다.
4. 전일 및 과거 일자의 결과를 재조회 가능하게 한다.

## 2-2. 성공 지표(KPI)

| 구분               | 지표                                   | 목표                       |
| ------------------ | -------------------------------------- | -------------------------- |
| 데이터 수집 안정성 | 일일 배치 성공률                       | 95% 이상                   |
| 데이터 완전성      | 시장별 대표 지수 수집 성공률           | 99% 이상                   |
| 뉴스 수집 품질     | 일일 시장별 정제 후 기사 수            | 최소 10건 이상             |
| 중복 제거 성능     | 중복 기사 비율 감소                    | 원본 대비 30% 이상 감소    |
| 페이지 생성 품질   | 시장별 일간 페이지 생성 성공률         | 95% 이상                   |
| 사용자 효율        | 한 페이지에서 핵심 뉴스 파악 가능 여부 | 정성 평가 기준 “충분” 이상 |
| 응답 성능          | 일간 페이지 조회 API 응답 시간         | 2초 이내                   |
| 운영 편의성        | 배치 실패 원인 확인 가능 여부          | 실패 로그 100% 기록        |

## 2-3. 정성적 성공 기준

- 사용자가 별도 뉴스 검색 없이 한 페이지에서 시장 분위기를 파악할 수 있어야 한다.
- 디자이너가 핵심 뉴스 카드, 지수 카드, 요약 영역의 우선순위를 명확히 이해할 수 있어야 한다.
- 개발자가 배치/저장/조회 구조를 분리된 책임으로 이해하고 구현할 수 있어야 한다.

| 연령대    | 30대 ~ 40대                                              |

# 4. User Journey Map

## 4-1. 주요 사용자 여정

| 단계              | 사용자 행동                                | 시스템 동작                         | 기대 결과                       |

## 5-2. 기능 목록

### FR-01. 시장별 일간 배치 실행 API

**우선순위:** P0

| 항목      | 내용                                                                            |
| --------- | ------------------------------------------------------------------------------- |
| 설명      | 미국/한국 시장에 대해 일간 데이터 수집 및 페이지 생성 배치를 실행한다           |
| 입력      | market_type(US/KR), business_date(optional), force(optional)                    |
| 출력      | 배치 실행 결과 ID, 상태, 대상 날짜                                              |
| 성공 조건 | 배치가 중복 실행되지 않고 성공/실패 상태를 저장해야 한다                        |
| 예외      | 동일 market_type + business_date에 대해 중복 실행 시 차단 또는 재생성 정책 적용 |

**세부 요구사항**

- API 예시: `POST /api/v1/batch/market-daily`
- `business_date` 미입력 시 시장별 기본 기준일 계산
- `force=true`일 경우 기존 페이지 재생성 허용 가능
- 실행 이력은 별도 테이블에 저장

### FR-02. 뉴스 수집 기능

**우선순위:** P0

| 항목      | 내용                                                           |
| --------- | -------------------------------------------------------------- |
| 설명      | Naver 뉴스 검색 API를 통해 시장 관련 뉴스 검색 결과를 수집한다 |
| 대상 시장 | 미국, 한국                                                     |
| 검색 대상 | 대표 지수 및 시장 관련 검색어 그룹                             |
| 저장 단위 | 검색 결과 원본 메타데이터 단위                                 |

**세부 요구사항**

- 미국 시장 검색어 그룹 예시
  - 나스닥
  - S&P 500
  - 다우존스
  - 미국 증시
  - 뉴욕증시
  - 미 증시 마감

- 한국 시장 검색어 그룹 예시
  - 코스피
  - 코스닥
  - 국내 증시
  - 한국 증시
  - 증시 마감

- 검색 결과에서 최소 아래 항목 저장
  - search_keyword
  - search_group
  - title
  - link
  - originallink
  - description
  - pub_date
  - fetched_at
  - market_type
  - business_date

**예외 처리**

- API 응답 실패 시 재시도 최대 3회
- 일부 검색어 실패 시 전체 배치를 중단하지 않고 실패 로그 누적
- 수집 결과 0건일 경우 상태를 Warning으로 기록

### FR-03. 대표 지수 수집 기능

**우선순위:** P0

| 항목      | 내용                                               |
| --------- | -------------------------------------------------- |
| 설명      | 시장별 대표 지수 데이터를 외부 소스로부터 수집한다 |
| 미국 지수 | 나스닥, 다우존스, S&P 500                          |
| 한국 지수 | 코스피, 코스닥                                     |

**세부 요구사항**

- 저장 항목
  - symbol

  - market_type
  - business_date
  - open_price
  - high_price
  - low_price
  - close_price
  - change_value
  - change_percent
  - fetched_at
  - provider_name

- provider를 교체 가능하도록 추상화 구조 적용
- 지수 수집 실패 시 배치 실패 여부는 정책으로 분리
  - 최소 1개 이상 지수 수집 실패 시 Warning

### FR-04. 뉴스 중복 제거 기능

**우선순위:** P0

| 항목 | 내용                                                |
| ---- | --------------------------------------------------- |
| 설명 | 동일 또는 유사 기사들을 식별하여 중복 노출을 줄인다 |
| 목적 | AI 비용 절감 및 페이지 품질 향상                    |
| 기준 | originallink, 제목 정규화, 유사도 비교              |

**세부 요구사항**

- 1차 중복 제거
  - `originallink` 동일 시 중복 처리

- 2차 중복 제거
  - 제목 정규화 후 동일 시 중복 처리

- 3차 유사 기사 클러스터링
  - 제목 또는 description 기준 유사 기사 그룹화

- 중복 제거 전/후 기사 수 기록 필수

### FR-06. AI 요약 생성 기능

**우선순위:** P0

| 항목      | 내용                                                         |
| --------- | ------------------------------------------------------------ |
| 설명      | 수집·정제된 뉴스와 지수 정보를 바탕으로 요약 정보를 생성한다 |
| 범위      | 클러스터 요약, 시장 전체 요약                                |
| 호출 단위 | 기사 개별 요약이 아닌 이슈 묶음 기반 요약 우선               |

**세부 요구사항**

- 생성 대상
  - 시장 한줄 요약
  - 핵심 뉴스 카드 요약
  - 시장 해설
  - 다음 관전 포인트

- 입력 데이터
  - 대표 지수 데이터
  - 정제된 뉴스 클러스터
  - 기사 제목/description/링크

- 출력 데이터
  - summary_short
  - summary_long
  - key_points
  - tags
  - outlook

- 실패 시 fallback
  - description 기반 간단 요약 또는 요약 미생성 표시

### FR-07. 일간 페이지 스냅샷 생성

# 7. User Interface (UI) Strategy

## 7-1. UI 목표

UI는 “많은 정보를 보여주는 것”이 아니라, **짧은 시간 안에 시장의 핵심을 파악하게 하는 것**을 목표로 한다.

## 7-2. 핵심 UX 원칙

- 상단에서 전체 분위기를 먼저 보여준다.
- 숫자(지수)와 텍스트(뉴스)를 분리하되 한 화면에서 자연스럽게 연결한다.
- 뉴스는 기사 목록이 아니라 “핵심 이슈 카드” 중심으로 보여준다.
- 원문 링크는 보조 기능으로 제공하되 접근은 쉬워야 한다.
- 빈 데이터, 누락 데이터, 실패 데이터는 숨기지 않고 명확히 상태를 보여준다.

## 7-3. 주요 화면 구성

### 화면 A. 메인/최신 시장 페이지

**목적:** 최신 시장 결과를 가장 빠르게 전달

구성 순서:

1. 페이지 헤더
2. 시장 요약 문장
3. 대표 지수 카드 영역
4. 오늘의 핵심 뉴스 카드 목록
5. 시장 해설
6. 관련 기사 링크 목록
7. 메타 정보

### 화면 B. 날짜별 아카이브 페이지

**목적:** 과거 시장 페이지 탐색

구성 요소:

- 시장 선택 탭: 미국 / 한국
- 날짜 선택 UI
- 날짜별 페이지 목록 또는 달력형 탐색

### 화면 C. 운영용 배치 상태 페이지

**목적:** 개인 운영 관점의 배치 확인

구성 요소:

- 최근 배치 상태
- 실패 이력
- 재실행 버튼(후속 확장 가능)

## 7-5. 디자이너를 위한 명시적 가이드

- 지수 카드와 뉴스 카드는 동일한 시각적 무게를 가지면 안 된다. 상단에서는 지수보다 **전체 요약 문장**이 먼저 인지되어야 한다.
- 뉴스 카드는 기사 개별 나열이 아니라 **클러스터 단위 카드**로 설계해야 한다.
- 관련 기사 수는 뱃지 형태로 간단히 표현 가능하다.
- 원문 링크는 새 탭 열기를 기본으로 한다.
- 데이터 누락 시 “정보 없음”이 아니라 구체적 상태 문구를 사용한다.
  - 예: “일부 지수 데이터 수집 실패”
  - 예: “AI 요약 미생성, 원문 기사만 제공”

# 8. Constraints & Risks

## 8-1. 기술적 제약

| 항목               | 설명                                                                             |
| ------------------ | -------------------------------------------------------------------------------- |
| 뉴스 데이터 범위   | Naver 검색 API 결과에 의존하므로 전체 시장 뉴스를 완전하게 커버하지 못할 수 있음 |
| 지수 데이터 제공자 | Yahoo Finance 접근 방식은 변경 가능성이 있으므로 provider 분리 필요              |
| 저장소             | MariaDB 기반이므로 대규모 검색/유사도 분석에는 한계가 있을 수 있음               |
| 배치 구조          | cron + curl 방식은 단순하지만 고급 재시도/분산 실행 제어에는 한계가 있음         |

## 8-2. 제품 리스크

| 리스크              | 설명                                               | 대응 방안                            |
| ------------------- | -------------------------------------------------- | ------------------------------------ |
| 검색 결과 품질 저하 | 시장 키워드가 너무 넓어 관련성 낮은 기사 포함 가능 | 검색어 그룹 튜닝, 필터링 규칙 추가   |
| 중복 기사 과다      | 동일 이슈 기사 과다 노출 가능                      | 링크/제목/유사도 기반 중복 제거      |
| AI 비용 증가        | 기사 수가 많을수록 비용 증가                       | 클러스터 단위 요약, 재요약 방지      |
| 날짜 혼동           | 미국 시장 날짜와 한국 실행 날짜 불일치             | business_date 분리 저장              |
| 일부 데이터 누락    | 외부 API 실패 시 페이지 품질 저하                  | 부분 실패 허용, 상태 표시            |
| 링크 깨짐           | 기사 원문 링크가 시간이 지나며 깨질 수 있음        | 네이버 링크/원문 링크 동시 저장 고려 |

## 8-3. 운영 리스크

- 외부 API rate limit 또는 응답 실패
- cronjob 중복 호출
- 동일 날짜 페이지 재생성 정책 혼선
- AI 출력 품질 편차
- 검색어 변경 시 과거 데이터와 형식 불일치

## 8-4. 반드시 합의되어야 할 정책

아래 항목은 개발 전 확정 필요하다.

| 정책 항목               | 결정 필요 내용                                  |
| ----------------------- | ----------------------------------------------- |
| business_date 산정 기준 | 미국/한국 시장별 기준 시각 정의                 |
| 재생성 정책             | 기존 페이지 덮어쓰기 vs version 증가            |
| 중복 기사 기준          | 유사도 임계값 및 대표 기사 선정 기준            |
| AI 실패 정책            | fallback 문구 노출 방식                         |
| 페이지 공개 범위        | 내부 개인용 고정인지 향후 공개 가능성 고려 여부 |
