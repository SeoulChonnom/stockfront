# PRD

**프로젝트명:** Market Daily Brief  
**문서 버전:** v0.2 Current UI Aligned  
**문서 목적:** 현재 구현되어 있는 Frontend 디자인과 라우트 구조를 기준으로, 개발자와 디자이너가 동일한 제품 범위와 데이터 요구사항을 이해하도록 정의한다.

---

# 1. Project Overview

## 1-1. 프로젝트 배경

사용자는 미국/한국 시장 흐름을 파악하기 위해 여러 뉴스 기사, 지수 데이터, 요약 콘텐츠를 따로 확인해야 한다. 이 과정은 시간이 오래 걸리고, 동일 이슈의 중복 기사 때문에 핵심 맥락을 빠르게 이해하기 어렵다.

본 프로젝트는 미국/한국 시장 결과를 한 화면에 통합하여 보여주는 일간 브리핑 웹 애플리케이션을 구축하는 것을 목표로 한다. 사용자는 최신 시장 화면에서 두 시장의 흐름을 한 번에 비교하고, 특정 날짜의 기록을 아카이브에서 찾아 다시 확인하고, 특정 이슈를 클러스터 상세 화면에서 더 깊게 볼 수 있어야 한다.

## 1-2. 해결하려는 문제

- 미국/한국 시장 뉴스를 여러 사이트에서 직접 검색해야 한다.
- 동일 이슈가 여러 기사로 반복 보도되어 핵심 이슈 파악이 어렵다.
- 대표 지수, 시장 해설, 관련 뉴스, 원문 링크가 분산되어 있어 복기가 불편하다.
- 특정 날짜의 시장 분위기를 다시 확인하려 할 때 탐색 경로가 길다.
- 운영자가 배치 실패와 품질 상태를 빠르게 파악하기 어렵다.

## 1-3. 현재 제품 정의

현재 제품은 다음 5개 화면으로 구성된다.

| 화면                      | 라우트                          | 설명                           |
| ------------------------- | ------------------------------- | ------------------------------ |
| 최신 시장 페이지          | `/market/latest`                | 미국/한국 통합 일간 시장 요약  |
| 날짜별 시장 페이지        | `/market/archive/:businessDate` | 특정 날짜의 통합 시장 요약     |
| 아카이브 검색 페이지      | `/market/archive/search`        | 날짜 범위/상태 기반 기록 탐색  |
| 뉴스 클러스터 상세 페이지 | `/market/cluster/:clusterId`    | 특정 이슈 심층 상세            |
| 배치 상태 페이지          | `/ops/batches`                  | 운영용 배치 상태/이력 모니터링 |

추가 동작:

- `/` 진입 시 `/market/latest`로 리다이렉트한다.
- 최신/날짜별 화면은 모두 미국+한국을 한 페이지 안에 함께 보여준다.
- 아카이브 검색 결과는 클러스터 상세가 아니라 날짜별 시장 페이지로 이동한다.

## 1-4. 프로젝트 범위

| 구분        | 범위 포함                                                                          | 범위 제외                         |
| ----------- | ---------------------------------------------------------------------------------- | --------------------------------- |
| 시장 범위   | 미국, 한국                                                                         | 추가 국가/자산군 확장             |
| 페이지 유형 | 최신 통합 일간 페이지, 날짜별 통합 페이지, 아카이브 검색, 클러스터 상세, 배치 상태 | 실시간 뉴스 스트림, 개인화 피드   |
| 뉴스 수집   | 시장 관련 뉴스 수집 및 정제                                                        | 개별 종목 단위 심층 뉴스 탐색     |
| 지수 데이터 | 대표 지수 카드용 데이터                                                            | 실시간 차트, 호가, 개별 종목 티커 |
| 저장소      | 스냅샷 보존 가능한 관계형 저장소                                                   | 전문 검색엔진 도입                |
| 운영 기능   | 배치 실행, 이력 확인, 상세 로그 확인                                               | 복잡한 워크플로우 오케스트레이터  |
| 사용자 범위 | 내부/개인 사용 중심                                                                | 계정 기반 다중 사용자 서비스      |

## 1-5. 핵심 가치 제안

- 미국/한국 시장 흐름을 한 페이지에서 빠르게 파악 가능
- 대표 지수, 핵심 이슈, 해설, 원문 링크가 하나의 맥락으로 연결됨
- 과거 날짜 기록을 검색 후 즉시 재진입 가능
- 특정 클러스터 이슈를 대표 기사와 관련 기사 타임라인으로 깊게 탐색 가능
- 운영자가 배치 품질과 실패 사유를 한 화면에서 확인 가능

# 2. Goals & Success Metrics

## 2-1. 핵심 목표

1. 최신 시장 화면에서 미국/한국 시장 결과를 통합된 한 페이지로 제공한다.
2. 날짜별 시장 화면에서 특정 날짜의 통합 스냅샷을 동일한 레이아웃으로 재현한다.
3. 핵심 뉴스는 기사 개별 목록이 아니라 클러스터 카드 중심으로 제공한다.
4. 사용자는 아카이브 검색을 통해 날짜별 화면으로 빠르게 이동할 수 있어야 한다.
5. 운영자는 배치 상태 페이지에서 성공/실패/부분 실패를 즉시 파악할 수 있어야 한다.

## 2-2. 성공 지표(KPI)

| 구분               | 지표                                         | 목표                           |
| ------------------ | -------------------------------------------- | ------------------------------ |
| 데이터 수집 안정성 | 일일 배치 성공률                             | 95% 이상                       |
| 페이지 생성 품질   | 통합 일간 페이지 생성 성공률                 | 95% 이상                       |
| 데이터 완전성      | 대표 지수 카드 최소 충족률                   | 미국 3개, 한국 2개 이상 권장   |
| 뉴스 품질          | 핵심 뉴스 카드 생성 수                       | 시장별 2개 이상, 전체 4개 이상 |
| 탐색 효율          | 아카이브 검색 후 날짜별 페이지 재진입 성공   | 정성 평가 기준 “원활” 이상     |
| 응답 성능          | 최신/날짜별 페이지 조회 응답 시간            | 평균 2초 이내                  |
| 운영 편의성        | 실패 원인 및 부분 실패 메시지 확인 가능 여부 | 100%                           |

## 2-3. 정성적 성공 기준

- 사용자는 최신 시장 화면에서 10~20초 안에 두 시장의 핵심 분위기를 파악할 수 있어야 한다.
- 디자이너와 개발자는 “통합 시장 페이지”가 제품의 기본 단위라는 점을 동일하게 이해해야 한다.
- 백엔드 개발자는 화면 렌더링에 필요한 데이터가 페이지 단위 스냅샷으로 제공되어야 함을 이해해야 한다.

# 5. Functional Requirements

## 5-1. 기능 우선순위 정의

- `P0`: 반드시 구현되어야 하는 핵심 기능
- `P1`: 초기 버전에 포함하는 것이 바람직한 기능
- `P2`: 후속 개선 항목

## 5-2. 기능 목록

### FR-01. 통합 일간 페이지 조회

**우선순위:** P0

| 항목      | 내용                                                                      |
| --------- | ------------------------------------------------------------------------- |
| 설명      | 최신 또는 특정 날짜의 미국/한국 통합 시장 페이지를 조회한다               |
| 화면      | 최신 시장 페이지, 날짜별 시장 페이지                                      |
| 출력      | 헤더, 글로벌 헤드라인, 시장별 섹션, 기사 링크, 메타 정보                  |
| 성공 조건 | Frontend가 추가 조합 없이 한 번의 응답으로 화면을 렌더링할 수 있어야 한다 |

세부 요구사항:

- 최신 페이지 조회와 날짜별 페이지 조회는 동일한 데이터 모델을 사용한다.
- 응답에는 반드시 다음 정보가 포함되어야 한다.
  - `businessDate`
  - `generatedAt`
  - `status`
  - `globalHeadline`
  - `markets[]`
  - 각 시장의 `summaryTitle`, `summaryBody`, `indices[]`, `topClusters[]`
  - 페이지 하단용 `articleLinks`
  - 페이지 메타 정보
- 최신/날짜별 화면은 미국/한국을 따로 나누어 호출하지 않는다.

### FR-02. 날짜별 시장 페이지 재현

**우선순위:** P0

| 항목      | 내용                                                                  |
| --------- | --------------------------------------------------------------------- |
| 설명      | 특정 `businessDate`의 통합 시장 스냅샷을 동일한 레이아웃으로 재현한다 |
| 진입 경로 | 아카이브 검색 결과, 클러스터 상세 화면의 CTA                          |
| 성공 조건 | 최신 시장 화면과 동일한 구조를 날짜만 바꿔 보여줘야 한다              |

세부 요구사항:

- 라우트는 `/market/archive/:businessDate` 구조를 따른다.
- 데이터가 없으면 Empty State를 표시할 수 있는 오류 코드/메시지를 반환해야 한다.
- 과거 날짜여도 글로벌 헤드라인과 시장별 카드 구조는 유지한다.

### FR-03. 아카이브 검색

**우선순위:** P0

| 항목 | 내용                                                                |
| ---- | ------------------------------------------------------------------- |
| 설명 | 사용자가 날짜 범위와 상태 기준으로 과거 통합 일간 페이지를 검색한다 |
| 화면 | `/market/archive/search`                                            |
| 입력 | `fromDate`, `toDate`, `status`, `page`                              |
| 출력 | 날짜, 글로벌 헤드라인 미리보기, 상태, 생성 시각                     |

세부 요구사항:

- 검색 결과는 날짜별 시장 페이지 진입용 목록이다.
- 결과 행 또는 제목 링크 클릭 시 `/market/archive/:businessDate`로 이동한다.
- 상태 필터는 `READY`, `PARTIAL`, `FAILED`를 지원한다.
- 기본 페이지네이션이 가능해야 한다.

### FR-04. 뉴스 클러스터 상세 조회

**우선순위:** P0

| 항목 | 내용                                                          |
| ---- | ------------------------------------------------------------- |
| 설명 | 핵심 뉴스 카드 클릭 시 특정 클러스터의 심층 상세를 보여준다   |
| 화면 | `/market/cluster/:clusterId`                                  |
| 출력 | 제목, 태그, 분석 문단, 대표 기사, 관련 기사 리스트, 메타 정보 |

세부 요구사항:

- 응답에는 `businessDate`, `marketLabel`, `title`, `tags`, `analysis[]`, `representativeArticle`, `articles[]`, `lastUpdatedAt`이 포함되어야 한다.
- 관련 기사 리스트에는 기사 제목, 언론사, 발행 시각, 원문 링크, 네이버 링크를 제공한다.
- 화면 하단 CTA를 위해 같은 날짜 페이지로 돌아갈 수 있는 `businessDate` 문맥이 필요하다.

### FR-05. 핵심 뉴스 카드 및 대표 링크 제공

**우선순위:** P0

| 항목 | 내용                                                |
| ---- | --------------------------------------------------- |
| 설명 | 최신/날짜별 시장 화면에서 핵심 뉴스 카드를 제공한다 |
| 목적 | 기사 개별 나열이 아니라 이슈 단위 소비를 지원       |
| 출력 | 대표 제목, 요약, 관련 기사 수, 태그, 상세 진입 정보 |

세부 요구사항:

- 시장별로 2개 이상의 클러스터 카드를 제공하는 것을 기본으로 한다.
- 각 카드에는 `clusterId`, `title`, `summary`, `articleCount`, `tags`가 포함되어야 한다.
- 대표 기사 링크는 카드 응답 내부에 포함하거나, 카드 클릭 후 상세 화면에서 안정적으로 제공되어야 한다.

### FR-06. 대표 지수 카드 제공

**우선순위:** P0

| 항목 | 내용                                                     |
| ---- | -------------------------------------------------------- |
| 설명 | 시장별 대표 지수 카드를 제공한다                         |
| 미국 | NASDAQ, S&P 500, DOW JONES 권장                          |
| 한국 | KOSPI, KOSDAQ 및 보조 지표 권장                          |
| 출력 | 지수명, 종가, 등락값, 등락률, 고가, 저가, 상승/하락 상태 |

세부 요구사항:

- Frontend는 각 시장의 카드 수를 그대로 렌더링하므로 응답이 카드 단위 배열이어야 한다.
- 숫자 값은 가공 가능한 원본 숫자 또는 화면 친화 문자열 중 정책을 정해 일관되게 제공한다.

### FR-07. 통합 기사 링크 목록 제공

**우선순위:** P0

| 항목 | 내용                                                |
| ---- | --------------------------------------------------- |
| 설명 | 일간 페이지 하단에서 원문 기사 링크를 제공한다      |
| 목적 | 요약의 출처 검증 및 심화 탐색                       |
| 출력 | 기사 제목, 언론사명, 발행 시각, 연결 클러스터, 링크 |

세부 요구사항:

- 페이지 응답 안에 별도 `articleLinks[]` 배열로 제공한다.
- 사용자 화면 기준으로 기사 링크를 위해 추가 API 호출이 발생하지 않아야 한다.

### FR-08. 통합 일간 배치 실행

**우선순위:** P0

| 항목 | 내용                                            |
| ---- | ----------------------------------------------- |
| 설명 | 특정 날짜의 미국/한국 통합 일간 배치를 실행한다 |
| 입력 | `businessDate`, `force`, `rebuildPageOnly`      |
| 출력 | `jobId`, 상태, 시작 시각                        |

세부 요구사항:

- 배치 단위는 더 이상 시장별 개별 실행이 아니라 날짜 기준 통합 실행이다.
- 중복 실행 방지 기준은 동일 `businessDate`다.
- 재생성 정책은 `force` 또는 `rebuildPageOnly`로 제어한다.

### FR-09. 배치 상태 목록 및 상세 조회

**우선순위:** P0

| 항목 | 내용                                                                      |
| ---- | ------------------------------------------------------------------------- |
| 설명 | 운영자가 최근 배치 상태와 선택한 배치의 상세 로그를 확인할 수 있어야 한다 |
| 화면 | `/ops/batches`                                                            |
| 출력 | 목록 테이블, 요약 통계, 선택 상세 패널                                    |

세부 요구사항:

- 목록 응답에는 `jobId`, `businessDate`, `status`, `startedAt`, `endedAt`, `duration`, `rawNewsCount`, `processedNewsCount`, `clusterCount`, `partialMessage`가 포함되어야 한다.
- 화면 상단 통계 카드 렌더링을 위해 성공/실패/부분 실패 집계 또는 이에 준하는 요약 정보가 필요하다.
- 상세 응답에는 에러 메시지 또는 로그 요약, 페이지 버전 정보가 포함되어야 한다.

### FR-10. 뉴스 수집 및 정제

**우선순위:** P0

| 항목 | 내용                                             |
| ---- | ------------------------------------------------ |
| 설명 | 미국/한국 시장 관련 뉴스를 수집하고 정제한다     |
| 목적 | 통합 페이지와 클러스터 상세에 필요한 콘텐츠 제공 |

세부 요구사항:

- 미국/한국 시장 관련 검색어 그룹을 기반으로 뉴스를 수집한다.
- 중복 제거와 클러스터링을 거쳐 카드 단위로 소비 가능한 구조를 만든다.
- 수집 결과 0건 또는 핵심 데이터 부족 시 `PARTIAL` 또는 `FAILED` 정책을 적용한다.

### FR-11. 대표 지수 수집

**우선순위:** P0

| 항목 | 내용                                                       |
| ---- | ---------------------------------------------------------- |
| 설명 | 최신/날짜별 시장 화면에 필요한 대표 지수 데이터를 수집한다 |
| 목적 | 시장 요약 카드 렌더링                                      |

세부 요구사항:

- 미국/한국 지수는 페이지에 필요한 최소 카드 수를 충족하도록 관리한다.
- 일부 지수 누락 시 전체 실패보다 부분 실패를 우선 고려한다.

### FR-12. AI 요약 생성

**우선순위:** P0

| 항목 | 내용                                                                 |
| ---- | -------------------------------------------------------------------- |
| 설명 | 뉴스 클러스터와 지수 데이터를 기반으로 시장/클러스터 요약을 생성한다 |
| 범위 | 글로벌 헤드라인, 시장 요약, 클러스터 카드 요약, 상세 분석            |

- 생성 대상

  - 시장 섹션 제목/요약
  - 클러스터 카드 요약
  - 클러스터 상세 분석
- 요약 실패 시 전체 실패로 처리하지 않고 fallback 또는 부분 실패 메시지를 허용한다.

### FR-13. 통합 스냅샷 저장

세부 요구사항:

- 저장 단위는 `businessDate + versionNo` 기반으로 관리한다.
- 스냅샷에는 화면 렌더링용 표현 데이터가 포함되어야 한다.
- 저장 항목 예시
  - `pageTitle`
  - `businessDate`
  - `status`
  - `globalHeadline`
  - `marketsJson`
  - `articleLinksJson`
  - `metadataJson`
  - `generatedAt`
  - `versionNo`

### FR-14. 부분 실패 허용형 렌더링

# 7. User Interface Strategy

## 7-1. UI 목표

UI는 짧은 시간 안에 시장 핵심을 파악하게 하는 것을 목표로 한다. 최신/날짜별 화면 모두 같은 레이아웃을 사용해 사용자가 “오늘”과 “과거”를 같은 방식으로 비교할 수 있어야 한다.

## 7-2. 핵심 UX 원칙

- 상단에서 전체 분위기를 먼저 보여준다.
- 미국/한국 시장을 하나의 페이지 안에 함께 배치한다.
- 지수와 뉴스는 분리하되 흐름은 자연스럽게 이어지게 한다.
- 뉴스는 기사 목록보다 클러스터 카드 중심으로 보여준다.
- 상태(`READY`, `PARTIAL`, `FAILED`)는 숨기지 않고 명확히 노출한다.

## 7-3. 주요 화면 구성

### 화면 A. 최신 시장 페이지

목적: 최신 통합 시장 결과를 가장 빠르게 전달

구성 순서:

1. 페이지 헤더
2. 글로벌 헤드라인
3. 미국 시장 섹션
4. 한국 시장 섹션

미국/한국 섹션 내부 구성:

1. 섹션 헤더
2. 시장 요약 카드
3. 대표 지수 카드 그리드
4. 핵심 뉴스 카드 그리드

### 화면 B. 날짜별 시장 페이지

목적: 특정 날짜의 통합 시장 결과를 동일한 구조로 확인

구성:

- 최신 시장 페이지와 동일한 본문 구조
- 헤더의 날짜 컨텍스트만 달라짐

### 화면 C. 아카이브 검색 페이지

목적: 과거 통합 시장 페이지 탐색

구성 요소:

- 날짜 범위 필터
- 상태 필터
- 검색 버튼
- 결과 테이블
- 페이지네이션

### 화면 D. 뉴스 클러스터 상세 페이지

목적: 특정 이슈의 심층 분석과 관련 기사 탐색

구성 요소:

- 브레드크럼
- 제목/태그
- AI 심층 분석 리포트
- 관련 뉴스 타임라인
- 대표 기사 카드
- 메타 정보
- 이전/같은 날짜 페이지 이동 CTA

### 화면 E. 운영용 배치 상태 페이지

목적: 배치 품질과 실패 상태 확인

구성 요소:

- 페이지 헤더
- 수치형 통계 카드
- 필터 바
- 배치 이력 테이블
- 선택 배치 상세 패널
- 수동 실행 버튼

## 7-4. 주요 컴포넌트 정의

# 8. Constraints & Risks

## 8-1. 기술적 제약

| 항목               | 설명                                                                            |
| ------------------ | ------------------------------------------------------------------------------- |
| 뉴스 데이터 범위   | 외부 뉴스 검색 API에 의존하므로 전체 시장 이슈를 완전하게 커버하지 못할 수 있음 |
| 지수 데이터 제공자 | 제공 방식 변경 가능성이 있어 provider 분리 필요                                 |
| 저장소             | 관계형 저장소 기반이므로 고급 검색/추천에는 한계가 있음                         |
| 배치 구조          | 단순 스케줄링은 구현이 쉽지만 고급 재시도/분산 제어에는 한계가 있음             |

## 8-2. 제품 리스크

| 리스크              | 설명                                                   | 대응 방안                         |
| ------------------- | ------------------------------------------------------ | --------------------------------- |
| 통합 페이지 과밀도  | 미국/한국을 함께 보여주므로 정보 밀도가 높아질 수 있음 | 헤더 우선순위와 섹션 구분 유지    |
| 검색 결과 품질 저하 | 시장 키워드가 넓어 관련성 낮은 기사 포함 가능          | 검색어 튜닝과 필터 규칙 추가      |
| 중복 기사 과다      | 동일 이슈 기사 과다 노출 가능                          | 링크/제목/유사도 기반 중복 제거   |
| AI 비용 증가        | 기사 수가 많을수록 비용 증가                           | 클러스터 단위 요약, 스냅샷 재사용 |
| 일부 데이터 누락    | 외부 API 실패 시 페이지 품질 저하                      | `PARTIAL` 상태와 누락 메시지 표시 |
| 링크 깨짐           | 시간이 지나며 원문 링크가 깨질 수 있음                 | 원문/미러 링크 병행 저장 고려     |

## 8-3. 운영 리스크

- 외부 API rate limit 또는 응답 실패
- 동일 날짜 배치 중복 실행
- 통합 스냅샷 재생성 정책 혼선
- AI 출력 품질 편차
- 과거 버전 유지 정책 미정으로 인한 데이터 혼선

## 8-4. 반드시 합의되어야 할 정책

| 정책 항목                | 결정 내용                                                                        |
| ------------------------ | -------------------------------------------------------------------------------- |
| `businessDate` 산정 기준 | 무조건 한국 시간(UTC+9) 기준 날짜를 사용한다.                                    |
| 재생성 정책              | 히스토리 보존을 위해 기존 페이지를 덮어쓰지 않고 항상 새 `versionNo`를 생성한다. |
| 대표 기사 선정 기준      | 대표 기사는 배치 시점에 대표 기사 ID를 확정 저장한다.                            |
| AI 실패 정책             | fallback 문구 노출 및 `PARTIAL` 처리 기준은 추가 합의가 필요하다.                |
| 운영 페이지 접근 범위    | 내부 전용 고정 여부는 추가 합의가 필요하다.                                      |

추가 합의 사항:

- 인증은 별도 서비스의 JWT를 사용하되, 동일 스키마를 사용하는 전제로 배치/운영 로그에는 JWT에서 추출한 `user_id`를 저장한다.
